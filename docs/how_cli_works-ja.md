# 🧠 How It Works: スタンドアロンCLIツール (Standalone Generator) の動作原理

[English](how_cli_works-en.md) | [한국어](how_cli_works.md) | [日本語](how_cli_works-ja.md) | [简体中文](how_cli_works-zh.md)

本ドキュメントは、`generate-tizen-app` プロジェクトで提供する **独立したCLIツール (`scripts/Generate-App.js`)** が大規模言語モデル(LLM)を活用して、自然言語の要件を実際のTizen C#アプリとして完璧にコンパイルされるコードに変換する仕組みを説明します。

*（エージェント環境での相互作用の仕組みについては、`how_agent_works.md` を参照してください。）*

---

## 🏗️ 1. エージェントの知識構造（知識のハイブリッド結合）

アプリのコードを生成する際、エージェント（LLM）は大きく2つの異なる知識ソースを結合してコードを作成します。

### A. Tizen.UI 専用知識 (注入された最新知識)
- **対象**: Tizen の最新オープンソース UI フレームワーク (`Tizen.UI`, `Tizen.UI.Components.Material` など)
- **ソース**: `ApiInfo/TizenUI_ControlCatalog.json` (Phase 1 において DLL から自動抽出されたメタデータ)
- **動作方式**: CLI はプロンプト作成時に **コンテキスト** として、Markdown に変換されたカタログ情報 (`TizenUI_ControlCatalog.md`) を一度に注入します。
- **効果**: フレームワークがアップデートされても、スクリプト (`Generate-ControlCatalog.js`) 経由でメタデータを再抽出して LLM に読み込ませれば、**常に最新の UI コンポーネント (`View`, `Button`, `VideoOverlayView` など) を幻覚 (Hallucination) ゼロで、正確な名前とプロパティを使用して生成** できます。

### B. Tizen.FX および C# 一般知識 (事前学習された内蔵知識)
- **対象**: ビデオ再生 (`Tizen.Multimedia`)、センサー、ネットワーク、Bluetooth などの **システム・コア機能 (Tizen.FX)** および C# の言語構文。
- **ソース**: 大規模言語モデル (LLM) 自身がすでに世界中の膨大なコードベースを通じて学習済みの「事前学習データ (Pre-trained Knowledge)」。
- **動作方式**: LLM は、「ビデオを再生するアプリを作成」というリクエストを受け取ると、あらかじめ学習している `Tizen.Multimedia.Player` クラスや、動画のロード・再生のシーケンス (`PrepareAsync()`, `Start()`) を優先して構成します。カタログにその内容が含まれていなくても、LLM の高度な推論能力によって結合させることができます。
- **効果**: カタログにすべての Tizen システム API を組み込まなくても、**UI 以外の機能については LLM の強力な背景知識が自動的に補完してくれます。**

## 🛠️ 2. 全体コード生成パイプライン

入力例: "動画プレイヤーアプリの生成"

1. **知識の結合**:
   LLM は、入力された `TizenUI_ControlCatalog` から画面配置用の UI 要素 (`Scaffold`, `VStack`, `VideoOverlayView`, `Material.Button`) を選んだ後、内部の知識を使って Media API (`Tizen.Multimedia.Player`) を呼び出す C# コードを統合して設計します。
2. **コード生成 (`MainView.cs` の置き換え)**:
   ベースとなるプロジェクトテンプレートのエントリポイント (`App.cs`) は固定したまま、メイン画面とロジックを担当する `MainView.cs` クラスの `CreateContent()` メソッドの内部のみを動的に生成・置き換えます。
3. **プロジェクト組み立て (`Create-TizenProject.js`)**:
   生成された `MainView.cs` および他の Tizen プロジェクトのファイル構造 (`.csproj`, `tizen-manifest.xml`) を `output/{AppName}` ディレクトリに組み立てます。
4. **ビルド (`dotnet build`)**:
   そのフォルダ内でビルドコマンドを実行し、実行可能な Tizen アプリケーション (`.tpk` または .NET ライブラリ) として抽出します。

## 🛡️ 3. 発生する可能性のあるリスクと自己修復(Self-Healing)メカニズム

LLMの内蔵知識（Tizen.FXなど）を100%信頼することはできません。プラットフォームのバージョンがアップデートされて以前のパラメータが変更されていたり、注入されたカタログの内容をLLMが勘違いして誤ったプロパティを使用したりする場合があります。これを克服するために、**Self-Healing（自己修復）ループ** を活用します。

1. `dotnet build` 実行段階で **コンパイルエラー** が検出される。
2. エラーログのパース（例：`CS0117: プロパティなし`, `CS1061: メソッドなし`, `CS0246: クラス/名前空間なし` など）。
3. **ローカルAPIスキルの自動呼び出し（APIヒントの動的注入）**
   - パースされたエラーから問題となるコンポーネント名（例：`Button`）やプロパティ名（例：`SetColumn`）を抽出します。
   - CLI ツールがバックグラウンドで `node scripts/search-tizen-api.js` スクリプトを呼び出し、`api-index.json` の生データから実際の API 仕様を検索してヒントを準備します。
4. このログと**「動的に抽出された API ヒント」**を、元々生成された `MainView.cs` のコードと一緒に**再度 LLM へフィードバック**処理します。
5. LLM が以前のバージョン知識が原因でエラーを起こした箇所を認識し、提供された正確な API ヒントに基づいて、コードを **自ら修正および再ビルド** します（最大再試行回数まで繰り返す）。

このような **[最新UIカタログ (プロンプト)] + [基本エンジン (内蔵知識)] + [ビルド/エラー分析 (APIスキルによる修復)]** の3拍子が噛み合うことで、ユーザーがターミナルに介入しなくても、たった1行の自然言語から完璧に動作するTizenアプリが誕生します。
